The backstory

Versioning was always a pain

API versioning has always been a pain in Spring. There was no built-in support, which meant every team rolled their own solution. Some hardcoded /v1/ in request paths. Others wrote custom RequestCondition implementations. A few pulled in third-party libraries that were inevitably abandoned.

Every project did it differently. There was no standard, no convention, and no framework guidance. The result: inconsistent APIs, duplicated controller classes, and routing logic scattered across annotation values, request matchers, and configuration files.

Spring Framework 7 finally treats API versioning as a first-class concern. It adds a version attribute directly to the request mapping annotations, plus a configuration API for choosing how versions travel over the wire.

The old way

Manual versioning in paths

Without framework support, the most common approach was to bake the version directly into the URL path. This meant separate controller classes (or at least separate methods with duplicated paths), no shared routing logic, and version strings scattered as raw text throughout the codebase.

Before — version in path strings 
@RestController
@RequestMapping("/v1/users")
public class UserControllerV1 {

    @GetMapping
    public List<UserV1> getUsers() {
        // V1 implementation
        return userService.getAllUsersV1();
    }
}

@RestController
@RequestMapping("/v2/users")
public class UserControllerV2 {

    @GetMapping
    public List<UserV2> getUsers() {
        // V2 implementation — different shape
        return userService.getAllUsersV2();
    }
}

// Duplicate controllers, no shared routing,
// version scattered as raw text in annotations

The new way

The version attribute

Spring Framework 7 adds a version attribute to @GetMapping, @PostMapping, and all the other request mapping annotations. One controller, one path, multiple versions. The framework handles the routing.

After — framework-managed versioning 
@RestController
@RequestMapping("/users")
public class UserController {

    @GetMapping(version = "1.0")
    public List<UserV1> getUsersV1() {
        return userService.getAllUsersV1();
    }

    @GetMapping(version = "2.0")
    public List<UserV2> getUsersV2() {
        return userService.getAllUsersV2();
    }
}

No duplicate controllers. No version in the path string. The version attribute is a semantic declaration — the framework decides how it maps to the wire protocol based on your configuration.

Configuration

ApiVersionConfigurer

You configure the versioning strategy through WebMvcConfigurer. Choose how versions travel over the wire, declare supported versions, and set a default. The configurer supports four strategies: path segment, header, query parameter, and media type.

WebMvcConfigurer — versioning setup 
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void configureApiVersioning(ApiVersionConfigurer configurer) {
        configurer
            .usePathSegment(1)
            .addSupportedVersions("1.0", "2.0")
            .setDefaultVersion("1.0");
    }
}

The usePathSegment(1) call means the version is extracted from position 1 in the path (e.g., /v1/users). Swap it for useHeader(), useQueryParameter(), or useMediaType() to change the strategy without touching your controllers.

Wire protocols

Four strategies, one API

The version attribute on your controller stays the same. Only the configurer changes. Here are the four supported strategies:

🔗

Path Segment

Version embedded in the URL path at a configured position.

GET /v1/users
📩

Custom Header

Version sent as a request header, keeping the URL clean.

X-API-Version: 1
🔍

Query Parameter

Version as a query string parameter. Easy to test in browsers.

GET /users?version=1
📦

Media Type

Version negotiated through the Accept header using vendor types.

Accept: application/vnd.app.v1+json

Client side

RestClient, WebClient & HTTP interfaces

Built-in client support

  • ApiVersionInserter — a shared component used by RestClient, WebClient, and HTTP interface clients to automatically insert the version into outgoing requests, matching whatever strategy the server uses.
  • HTTP interface clients — the @HttpExchange-based declarative clients pick up version configuration automatically. No manual header or path manipulation.
  • Deprecation handling (RFC 9745) — Spring can signal that an API version is deprecated using the standard Deprecation HTTP header, so clients get a machine-readable deprecation notice.
ApiVersionInserter with RestClient 
ApiVersionInserter inserter = ApiVersionInserter
    .useHeader("X-API-Version");

RestClient client = RestClient.builder()
    .baseUrl("https://api.example.com")
    .apiVersionInserter(inserter)
    .build();

// Requests automatically include X-API-Version header
List<User> users = client.get()
    .uri("/users")
    .apiVersion("2.0")
    .retrieve()
    .body(new ParameterizedTypeReference<>() {});

Side by side

Before vs After

Before — Spring Boot 3 
// Two separate controllers
@RestController
@RequestMapping("/v1/users")
public class UserV1Controller {

    @GetMapping
    public List<UserV1> list() { ... }

    @GetMapping("/{id}")
    public UserV1 get(@PathVariable Long id)
        { ... }
}

@RestController
@RequestMapping("/v2/users")
public class UserV2Controller {

    @GetMapping
    public List<UserV2> list() { ... }

    @GetMapping("/{id}")
    public UserV2 get(@PathVariable Long id)
        { ... }
}

// Duplicated classes, paths, logic
// Version is just a string in the URL
// No framework awareness of versions
After — Spring Boot 4 
// One controller, clean paths
@RestController
@RequestMapping("/users")
public class UserController {

    @GetMapping(version = "1.0")
    public List<UserV1> listV1() { ... }

    @GetMapping(version = "2.0")
    public List<UserV2> listV2() { ... }

    @GetMapping(value = "/{id}",
               version = "1.0")
    public UserV1 getV1(@PathVariable Long id)
        { ... }

    @GetMapping(value = "/{id}",
               version = "2.0")
    public UserV2 getV2(@PathVariable Long id)
        { ... }
}

// Framework manages version routing
// Strategy configured in one place
// Deprecation signaled via RFC 9745
