The backstory

25 years of heavyweight threads

Since Java 1.0, every Thread has been a thin wrapper around an OS thread. Each one allocates ~1 MB of stack memory and requires a kernel-level context switch. That was fine in 1998 when a server handled dozens of requests. It's a problem now, when a single microservice might need to handle tens of thousands of concurrent connections.

The workarounds — thread pools, reactive programming, async/await patterns — add complexity without solving the root problem: platform threads are too expensive to represent every unit of concurrent work.

Project Loom, delivered in Java 21, introduces three new primitives that fix this from the ground up.

Part 1

Virtual Threads

Lightweight threads managed by the JVM, not the OS.

The Old Way: Platform Thread Pool

Your server has a fixed pool of platform threads (typically 200). Each incoming request claims one. When a thread blocks on I/O — a database query, an HTTP call — it sits idle. The thread is stuck, and so is every request waiting behind it.

The New Way: Virtual Threads

Virtual threads are managed by the JVM, not the OS. They're extremely cheap — you can create millions of them. When a virtual thread blocks on I/O, it unmounts from its carrier thread. The carrier immediately picks up another virtual thread. No thread ever sits idle.

The code is simpler, not harder

Virtual threads use the same Thread API you already know. No new abstractions, no reactive chains, no callbacks. Write blocking code — the JVM handles the rest.

Before — thread pool 
var pool = Executors.newFixedThreadPool(200);

for (var request : requests) {
    pool.submit(() -> {
        // blocks a platform thread
        var user = db.findUser(request.id());
        var orders = api.getOrders(user);
        return buildResponse(user, orders);
    });
}
After — virtual threads 
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {

    for (var request : requests) {
        executor.submit(() -> {
            // same blocking code, but on a virtual thread
            var user = db.findUser(request.id());
            var orders = api.getOrders(user);
            return buildResponse(user, orders);
        });
    }
}

The only change: newFixedThreadPool(200) becomes newVirtualThreadPerTaskExecutor(). No pool size to tune. No thread starvation. One virtual thread per task, as many as you need.

Part 2

Scoped Values

Immutable, scoped data sharing across threads — replacing ThreadLocal.

The Old Way: ThreadLocal

ThreadLocal stores per-thread data. When you spawn child threads, values get copied to each child. Those copies are independent — they can be mutated separately, leading to stale data. And with virtual threads creating thousands of threads, those copies add up fast.

The problem
ThreadLocal<User> = "Alice"
Child-1: "Alice" COPY
Child-2: "Alice" COPY
Child-3: "Alice" MUTATED
⚠ Child-3 mutated its copy — other threads still see "Alice". Who's right?

The New Way: ScopedValue

ScopedValue binds a value for the duration of a scope. It's immutable within that scope — no thread can change it. Child threads see the parent's value directly, no copying. When the scope ends, the value is gone — no cleanup, no leaks.

The solution
ScopedValue.where(USER, "Alice").run(() -> { ... })
USER = "Alice"   🔒
Thread-1 reads: "Alice"
Thread-2 reads: "Alice"
Thread-3 reads: "Alice"
✓ Immutable — no thread can modify the value   |   ✓ No copies — zero allocation overhead
Before — ThreadLocal 
static final ThreadLocal<User> CURRENT_USER =
    new ThreadLocal<>();

void handleRequest(Request req) {
    CURRENT_USER.set(req.user());
    try {
        processOrder();
        // any thread can call CURRENT_USER.set()
        // child threads get independent copies
        // forgetting remove() → memory leak
    } finally {
        CURRENT_USER.remove(); // easy to forget!
    }
}
After — ScopedValue 
static final ScopedValue<User> CURRENT_USER =
    ScopedValue.newInstance();

void handleRequest(Request req) {
    ScopedValue.where(CURRENT_USER, req.user())
        .run(() -> {
            processOrder();
            // immutable — can't be changed
            // child threads share the binding
            // auto-cleanup when scope exits
        });
}
Part 3

Structured Concurrency

Concurrent tasks as a single unit of work — no leaks, no orphans.

The Old Way: Fire and Forget

With ExecutorService, you submit tasks into the void. If one fails, you might not notice until later. If the parent is done but a child is still running, that's a thread leak. Cancellation? You have to do it yourself — and you'll probably miss some.

The problem
main() ❓
⚙ Task A OK
⚙ Task B FAILED
⚙ Task C LEAKED
⚠ Task B failed silently — Task C is still running, consuming resources — nobody is watching

The New Way: StructuredTaskScope

StructuredTaskScope treats concurrent tasks like a structured block: all forked tasks must complete before the scope exits. If one fails, the others are automatically cancelled. Errors propagate to the parent. No leaks. No orphans. The concurrent equivalent of a try-with-resources.

The solution
scope.join()
⚙ fork(A) OK
⚙ fork(B) FAILED
⚙ fork(C) CANCELLED
✓ Task B failed → Task C auto-cancelled → error propagated to parent → scope exited cleanly
Before — unstructured 
var pool = Executors.newFixedThreadPool(3);

Future<User>    userF = pool.submit(() -> fetchUser());
Future<Order>   orderF = pool.submit(() -> fetchOrder());
Future<Invoice> invoiceF = pool.submit(() -> fetchInvoice());

// if fetchOrder() fails...
// fetchUser() and fetchInvoice() keep running
// we have to manually cancel them
// and we might forget
var user = userF.get();     // blocks, ignoring error
var order = orderF.get();   // throws here!
var invoice = invoiceF.get(); // still running...
After — structured 
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

    Subtask<User>    user = scope.fork(() -> fetchUser());
    Subtask<Order>   order = scope.fork(() -> fetchOrder());
    Subtask<Invoice> invoice = scope.fork(() -> fetchInvoice());

    scope.join();            // waits for all
    scope.throwIfFailed();   // propagates first error
    // if fetchOrder() fails →
    // fetchUser() + fetchInvoice() auto-cancelled
    // scope exits, everything cleaned up

    return build(user.get(), order.get(), invoice.get());
}
Spring Boot

This isn't theoretical

If you use Spring Boot, virtual threads are already part of your stack.

⚙

Spring Boot & Virtual Threads

Spring Boot adopted virtual threads early. Every request handler, every @Async method, every scheduled task — all running on virtual threads. Your existing blocking code (JPA queries, REST calls, JDBC) works as-is, but now scales to thousands of concurrent requests without reactive complexity.

# application.properties
spring.threads.virtual.enabled=true

// Opt-in: one line of config
// Tomcat, Jetty, Undertow all supported

What this means for your code

What people did for scale 
// Reactive style — necessary evil for throughput
@GetMapping("/orders/{id}")
Mono<Order> getOrder(@PathVariable String id) {
    return userClient.findById(id)
        .flatMap(user ->
            orderClient.findByUser(user)
                .zipWith(invoiceClient.findByUser(user))
        )
        .map(tuple -> buildOrder(tuple));
    // Hard to read, hard to debug,
    // stack traces are useless
}
What you can write now 
// Blocking style — same throughput, virtual threads
@GetMapping("/orders/{id}")
Order getOrder(@PathVariable String id) {
    var user = userClient.findById(id);
    var order = orderClient.findByUser(user);
    var invoice = invoiceClient.findByUser(user);

    return buildOrder(user, order, invoice);
    // Easy to read, easy to debug,
    // full stack traces work
}

Same throughput. Readable code. Real stack traces. Debuggable. Testable. No reactive library needed for I/O-bound workloads.

Three primitives, one coherent model

They're designed to work together.

⇅

Virtual Threads

Make threads cheap enough to use one per task. No more pooling, no more reactive workarounds.

🔒

Scoped Values

Share context across those threads safely — immutable, zero-copy, auto-cleaned.

☍

Structured Concurrency

Keep those threads organized — parent-child relationships, automatic cancellation, no leaks.

The full picture

All three, together 
static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

@GetMapping("/dashboard")
Dashboard getDashboard(@AuthenticatedUser User user) {
    // Scoped Value: bind user for the entire request scope
    return ScopedValue.where(CURRENT_USER, user).call(() -> {

        // Structured Concurrency: fork subtasks, auto-cancel on failure
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {

            // Virtual Threads: each fork runs on its own virtual thread
            var orders   = scope.fork(() -> orderService.forUser(CURRENT_USER.get()));
            var invoices = scope.fork(() -> invoiceService.forUser(CURRENT_USER.get()));
            var prefs    = scope.fork(() -> prefService.forUser(CURRENT_USER.get()));

            scope.join();
            scope.throwIfFailed();

            return new Dashboard(orders.get(), invoices.get(), prefs.get());
        }
    });
}
