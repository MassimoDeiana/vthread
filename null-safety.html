The Problem

The Billion-Dollar Mistake

Tony Hoare called it his "billion-dollar mistake" — the invention of the null reference in 1965. In Java, NullPointerException is the single most common runtime error. It crashes applications in production, corrupts data, and wastes thousands of debugging hours.

The core issue: Java has no language-level null safety. Any reference can be null at any time. Teams defend against this with scattered null checks, Optional wrappers, and sheer hope — or they forget and crash in production at 3 AM.

The Old Way

Scattered Annotations, No Enforcement

Before JSpecify, the Java ecosystem had multiple competing null-annotation standards. Spring had its own @Nullable / @NonNull from org.springframework.lang. JSR-305 introduced javax.annotation.Nullable. JetBrains, FindBugs, and Eclipse each had their own. None were enforced at compile time.

Spring 6 — verbose @NonNull everywhere 
@Service
public class UserService {

    @NonNull
    private final UserRepository repository;

    public UserService(@NonNull UserRepository repository) {
        this.repository = repository;
    }

    @NonNull
    public User findById(@NonNull Long id) {
        return repository.findById(id)
            .orElseThrow(() -> new NotFoundException(id));
    }

    @Nullable
    public User findByEmail(@NonNull String email) {
        return repository.findByEmail(email).orElse(null);
    }
}

Every parameter and return type needs an annotation. Miss one and you have no safety at all. The noise drowns the signal.

The New Way

JSpecify: Non-Null by Default

JSpecify flips the model. Put @NullMarked on your package-info.java and everything in that package is non-null by default. Only annotate the exceptions with @Nullable. The result: dramatically less annotation noise, and the annotations that remain carry real meaning.

package-info.java — one annotation, package-wide 
@NullMarked
package com.example.app.service;

import org.jspecify.annotations.NullMarked;
UserService.java — clean, non-null by default 
@Service
public class UserService {

    private final UserRepository repository;  // non-null by default

    public UserService(UserRepository repository) {  // non-null param
        this.repository = repository;
    }

    public User findById(Long id) {  // returns non-null
        return repository.findById(id)
            .orElseThrow(() -> new NotFoundException(id));
    }

    public @Nullable User findByEmail(String email) {  // only this is nullable
        return repository.findByEmail(email).orElse(null);
    }
}

Only the exceptions need annotations. Everything else is non-null by default.

Important: @NullMarked does not cascade to sub-packages. Each package needs its own package-info.java.
Tooling

Compile-Time Enforcement, IDE Support, Cross-Language

JSpecify is not just annotations — it is a specification that tools understand natively. The ecosystem is converging on it as the single standard.

Annotations alone are just documentation. Without tooling, nothing actually stops you from passing null. You need at least one of these to get real enforcement.

⚠️

NullAway

Error Prone plugin that fails your build on null violations. This is the real enforcement layer — without it, @NullMarked is advisory only. Integrates with Gradle and Maven.

Build-time — errors
💻

IntelliJ 2025.3+

Native JSpecify support. Warnings and inspections in the editor — highlights violations as you type, suggests quick-fixes. Helpful, but won't fail your CI.

IDE — warnings
🌐

Kotlin 2

Auto-translates JSpecify annotations to Kotlin nullability. @Nullable String becomes String? seamlessly.

Cross-language

Third-party blind spot

JSpecify only helps where both sides of a call are annotated. If you call a library that hasn't adopted JSpecify — and most haven't yet — those types appear as unannotated. NullAway won't flag violations at the boundary. You're back to defensive checking for anything coming from outside your @NullMarked code.
